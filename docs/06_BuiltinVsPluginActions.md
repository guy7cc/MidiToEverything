# 06. 既定アクションの全DLL化（プラグイン化）に関する判断

> 問い: 既定（組み込み）アクションも統一してすべてプラグイン DLL にすべきか？
> **結論: しない。** 組み込みは第一級のまま、拡張はプラグインという二層モデルを維持する。

## 背景

現在の構成は二層:

- **組み込みアクション**（21種: Key/MouseClick/…/Obs/MidiOut/Macro/Toggle 等）
  — `InputAction` 判別共用体＋専用ハンドラ＋型付き DTO（`[JsonDerivedType]`）＋
  per-kind エディタテンプレート＋検証/候補/ヘルプ/インストラクション。
- **プラグイン（外部 DLL）** — `IActionPlugin.Execute(command, arg)` を実装し、
  `PluginAction(pluginId, command, arg)` でルーティング（docs/05 §10）。意図的に薄い拡張点。

## 「全DLL化」が要求するもの

組み込みアクションをプラグインに置き換えるには、プラグイン API が最低限:

1. **新しいアクション型の寄与** — 各プラグインが独自の型付きパラメータを持つ
   （Launch=対象+引数+作業ディレクトリ、Uia=ウィンドウ+要素+動作+値 …）。
2. **永続化の拡張** — `[JsonDerivedType]` の判別子を動的に開く必要がある。プラグイン由来の
   型を含む設定の読み書き・前方互換・未知型の扱いが複雑化。
3. **エディタ UI の寄与** — per-kind テンプレート・検証・候補・ヘルプ・インストラクション・
   専用ダイアログを、アセンブリ境界をまたいでプラグインが提供する必要がある。
4. **値連動/トリガー意味論** — Absolute フェーダー→値などの連続値処理の宣言。

これは「コマンド文字列を渡すだけ」の現行プラグイン契約とは桁違いに重い、
**フル機能のアクション SDK** を作ることに等しい。

## 判断の根拠

- **UX の劣化リスク**: 上記を諦めて組み込みを薄い `command/arg` 契約に落とすと、
  リッチなエディタ・型安全な永続化・検証・ヘルプをすべて失う。現在の使い勝手が後退する。
- **複雑性とバージョニング地獄**: UI/DTO/検証をアセンブリ境界で寄与させる設計は、
  ホストとプラグインの契約バージョン整合、信頼境界、クラッシュ分離など多大な負担を生む。
- **費用対効果**: 得られるのは「一貫性」という概念的価値のみ。大規模リファクタと
  回帰リスクに見合わない。
- **業界の通例**: VS Code（組み込み＋拡張）、OBS（組み込みソース＋プラグイン）、
  ブラウザ（コア＋拡張）等、成功例はいずれも **コアは第一級、拡張は別レイヤ** の二層。
  「すべて拡張」にしている例は稀で、保守性が低い。

## 維持する方針

- 組み込みアクション = 型付き・第一級（最良の UX・型安全・検証）。
- サードパーティ拡張 = `IActionPlugin` プラグイン（薄く安全な command/arg 契約）。
- 将来プラグインを強化する場合も、**追加の拡張点**（例: プラグインが候補/ヘルプ文字列を
  提供する軽量フック）に留め、組み込みの置き換えはしない。

## もし将来「全DLL化」を再検討するなら

最小でも次が前提条件: 安定したアクション・パラメータ記述子スキーマ、動的 DTO 多態の
安全な開放、エディタのメタ駆動フォーム（docs/05 §3.5 案B）、プラグインのクラッシュ分離。
これらが揃い、かつ第三者が高度アクションを多数提供する需要が顕在化した段階で再評価する。
現時点では時期尚早。
