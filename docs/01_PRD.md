# 要件定義書 (PRD) — MidiToEverything

> MIDI デバイスを汎用的な「左手デバイス（ショートカットパッド）」として利用するための Windows 向けデスクトップソフトウェア

- ドキュメントバージョン: 1.0
- 対象プラットフォーム: Windows 10 (2004+) / Windows 11
- 想定読者: 開発者（実装は Claude Code を併用）、レビュアー

---

## 1. 背景・目的

イラスト制作（Clip Studio Paint 等）、動画編集、3D/CAD、配信などの作業では、左手で大量のキーボードショートカットを使う。専用の左手デバイス（Tourbox / Loupedeck / 左手キーボード等）は高価で、ノブやフェーダーの数・割り当て自由度に制約がある。

一方、MIDI コントローラー（鍵盤、パッド、ノブ、フェーダー）は安価かつ豊富で、多数の物理操作子を備える。本ソフトウェアは、**任意の MIDI デバイスを OS レベルの汎用入力デバイス（キーボード/マウス）に変換する**ことで、ユーザーが自由にカスタマイズできる左手デバイス環境を提供する。

### ゴール
- 手持ちの MIDI コントローラーを、アプリごとに最適化したショートカットパッドとして使えるようにする。
- ノブ/フェーダーの連続値を「ブラシサイズ」「キャンバス回転」などの連続操作にマッピングできる。
- アクティブウィンドウに応じて自動でプロファイルを切り替え、ユーザーが意識せず最適な割り当てになる。

### 非ゴール（スコープ外 / 初期リリース）
- MIDI 出力（DAW へのフィードバック送信）。※将来拡張で検討
- macOS / Linux 対応。※アーキテクチャ上は分離するが初期はビルドしない
- クラウド同期・アカウント管理。
- マクロのスクリプト言語（Lua 等）による高度自動化。※将来拡張

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| MIDI イベント | MIDI デバイスから受信するメッセージ。Note On/Off, Control Change (CC), Pitch Bend, Program Change 等 |
| 入力シグナル (Signal) | マッピングの「キー」となる MIDI イベントの識別子。`(デバイス, チャンネル, メッセージ種別, 番号)` で一意化 |
| アクション (Action) | シグナルに対して OS に発行する操作。キー入力、マウスクリック、カーソル移動、プロファイル切替など |
| バインディング (Binding) | 1 つのシグナルと 1 つ以上のアクションの対応関係 |
| プロファイル (Profile) | バインディングの集合。アプリ単位で切り替える |
| 基本プロファイル (Base/Global Profile) | 全プロファイル・全ウィンドウで常に適用される最下層のプロファイル |
| コンテキスト | 現在フォアグラウンドにあるアクティブウィンドウのプロセス・タイトル情報 |
| 有効マッピング (Effective Map) | 基本プロファイルとアクティブプロファイルを合成して解決された、現時点で実際に発行されるマッピング |

## 3. ペルソナ / 主要ユースケース

### ペルソナ
- **イラストレーター A**: Clip Studio Paint をメインに使用。MIDI ノブでブラシサイズ、フェーダーで不透明度、パッドで取り消し/やり直しをしたい。
- **動画編集者 B**: DaVinci Resolve / Premiere を併用。アプリを切り替えるたびに割り当ても自動で変えたい。
- **配信者 C**: OBS のシーン切替・ミュートを MIDI パッドに割り当て、ゲーム中も使いたい。

### 主要ユースケース
1. MIDI デバイスを USB 接続すると自動認識され、入力モニターに反応が表示される。
2. 入力モニターを見ながら「このノブ＝Ctrl+回転」のようにバインディングを GUI で設定する。
3. Clip Studio Paint を前面にするとイラスト用プロファイルへ自動切替。ブラウザに移ると基本プロファイルのみ有効。
4. パッドの 1 つに「プロファイル手動切替」を割り当て、デバイス操作だけでプロファイルを巡回する。
5. すべてのアプリで共通の「Undo」「Copy」を基本プロファイルに置き、個別プロファイルで必要な箇所だけ上書きする。

## 4. 機能要件

要件 ID は `FR-x`。優先度: **MUST**（初期必須） / **SHOULD**（早期に欲しい） / **COULD**（あれば良い）。

### 4.1 MIDI デバイス管理
| ID | 要件 | 優先度 |
|----|------|--------|
| FR-1.1 | OS に接続済みの MIDI 入力デバイス一覧を列挙し UI に表示する | MUST |
| FR-1.2 | デバイスの接続/切断を動的（ホットプラグ）に検知し、UI と内部状態へ即時反映する | MUST |
| FR-1.3 | 複数デバイスの同時接続・同時監視に対応する | SHOULD |
| FR-1.4 | デバイス切断時、再接続したら自動で監視を再開する（デバイス名による同定） | SHOULD |
| FR-1.5 | 監視対象デバイスを個別に ON/OFF できる | COULD |

### 4.2 入力モニター
| ID | 要件 | 優先度 |
|----|------|--------|
| FR-2.1 | 受信した MIDI イベントを時系列ログでリアルタイム表示する（種別・チャンネル・番号・値・タイムスタンプ・デバイス名） | MUST |
| FR-2.2 | ログのフィルタリング（種別/チャンネル/デバイス別）と一時停止/クリアが可能 | SHOULD |
| FR-2.3 | 「最後に受信したシグナル」をワンクリックでバインディング対象として取り込める（ラーン機能） | MUST |
| FR-2.4 | 受信レートが高くても UI がフリーズしない（描画の間引き/バッチ更新） | MUST |

### 4.3 視覚的フィードバック
| ID | 要件 | 優先度 |
|----|------|--------|
| FR-3.1 | ノブ/フェーダー/キーの操作を GUI 上のインジケーターでリアルタイムに可視化する | MUST |
| FR-3.2 | CC/Pitch Bend は現在値をゲージ・ノブ表示、Note は押下ハイライトで表現する | MUST |
| FR-3.3 | バインディング済みのシグナルにはラベル（割当先アクション名）を重畳表示する | SHOULD |
| FR-3.4 | 未割り当ての操作子も自動で枠を生成して可視化する（汎用グリッド表示） | SHOULD |

### 4.4 マッピングエンジン
| ID | 要件 | 優先度 |
|----|------|--------|
| FR-4.1 | シグナルを OS のキーボードショートカット（修飾キー＋キー）に変換し発行する | MUST |
| FR-4.2 | シグナルをマウスクリック（左/右/中、ダブルクリック）に変換できる | MUST |
| FR-4.3 | シグナルをカーソル移動（相対/絶対）・ホイールスクロールに変換できる | MUST |
| FR-4.4 | CC/Pitch Bend の連続値を、相対加減算（エンコーダー）として扱える（例: 値の増減でズーム±） | MUST |
| FR-4.5 | Note On で押下、Note Off で離す「ホールド型」キー発行に対応（修飾キー押しっぱなし等） | MUST |
| FR-4.6 | 1 シグナルに複数アクションの順次実行（マクロ）を割り当てられる | SHOULD |
| FR-4.7 | 押下中のオートリピート（キーリピート）に対応 | COULD |
| FR-4.8 | 値域・しきい値・デッドゾーン・反転・感度（スケール）を設定できる | SHOULD |

### 4.5 プロファイル / コンテキスト
| ID | 要件 | 優先度 |
|----|------|--------|
| FR-5.1 | アクティブウィンドウ（フォアグラウンドプロセス）を常時監視する | MUST |
| FR-5.2 | プロファイルにマッチ条件（`"プロセス名\nタイトル"` に対する**単一の正規表現**）を設定し、一致時に自動切替する。エディタは起動中プロセスを候補として提示し、選択/入力したプロセスを正規表現へ自動統合（失敗時は通知）。正規表現は手動編集可 | MUST |
| FR-5.3 | UI からプロファイルを手動切替できる | MUST |
| FR-5.4 | 特定の MIDI シグナルにプロファイル切替アクション（指定/次/前/トグル）を割り当てられる | MUST |
| FR-5.5 | 手動切替を「ピン留め」して自動切替を一時的に無効化できる | SHOULD |
| FR-5.6 | 現在有効なプロファイルとコンテキストを UI とトレイで常時表示する | MUST |

### 4.6 基本プロファイル（継承・オーバーライド）
| ID | 要件 | 優先度 |
|----|------|--------|
| FR-6.1 | 全プロファイル・全ウィンドウで常時適用される基本プロファイルを 1 つ定義できる | MUST |
| FR-6.2 | 競合解決: 同一シグナルに対し、アクティブプロファイルの定義が基本プロファイルの定義を上書きする | MUST |
| FR-6.3 | アクティブプロファイルが定義しないシグナルは、基本プロファイルの定義へフォールバックする | MUST |
| FR-6.4 | 個別プロファイルから基本のバインディングを明示的に「無効化（ブロック）」できる | SHOULD |
| FR-6.5 | UI 上で、ある操作子の有効アクションが「基本由来」か「上書き」かを区別表示する | SHOULD |

### 4.7 永続化・運用
| ID | 要件 | 優先度 |
|----|------|--------|
| FR-7.1 | プロファイル/設定を人間可読な JSON で保存し、手動編集・共有が可能 | MUST |
| FR-7.2 | 設定変更を即時反映（再起動不要）する | MUST |
| FR-7.3 | システムトレイに常駐し、最小化してバックグラウンド動作する | MUST |
| FR-7.4 | プロファイルのインポート/エクスポート | SHOULD |
| FR-7.5 | スキーマバージョンを持ち、将来のマイグレーションに対応する | SHOULD |
| FR-7.6 | Windows ログオン時の自動起動オプション | COULD |

## 5. 非機能要件

| 分類 | 要件 |
|------|------|
| レイテンシ | MIDI 受信からキー/マウス発行までの追加遅延を **平均 < 10 ms / 最悪 < 30 ms** に抑える（人間が体感しない範囲） |
| スループット | 連続したノブ操作（数百 msg/秒/デバイス）でも取りこぼし・UI 凍結が起きない |
| 安定性 | 24 時間以上の常駐でメモリリーク・ハンドルリークがない |
| 応答性 | 入力イベント処理は UI スレッドをブロックしない（後述のスレッド分離） |
| 保守性 | レイヤ分割・DI・ユニットテスト可能なコア（マッピング解決ロジックは UI 非依存） |
| 安全性 | 入力発行は誤爆を避けるため、無効/未割当シグナルは確実に無視する。緊急停止（全発行停止）ホットキーを用意 |
| セキュリティ | キーストローク送信ソフトのため、設定ファイルはローカル限定・外部送信なし。管理者権限は原則不要（必要なケースは 7 章参照） |
| 配布 | 単一インストーラ or 自己完結型実行ファイル（.NET self-contained）で配布可能 |

## 6. 制約・前提

- Windows 標準の MIDI スタック（WinMM）を利用する。USB-MIDI クラスコンプライアントデバイスを前提とする。
- 一部の MIDI ドライバは「1 デバイス＝1 アプリ占有」となる場合があり、DAW と同時併用時は競合し得る（既知の制約として明記）。
- ゲームや一部アプリは仮想キー入力を受け付けない/スキャンコードを要求するため、**スキャンコード送信**を既定とする。
- 管理者権限で動作するアプリ（UAC 昇格済みウィンドウ）へキー送信するには、本ソフトも昇格が必要（UIPI 制約）。初期リリースでは「非昇格で動く範囲」を主対象とし、制約を明記する。

## 7. リスクと対応

| リスク | 影響 | 対応 |
|--------|------|------|
| MIDI ドライバのデバイス占有 | DAW 併用不可 | ドキュメント明記。将来 loopMIDI 経由の分岐を検討 |
| UIPI（昇格アプリへ送信不可） | 一部アプリで効かない | 起動時に管理者起動オプションを提供、警告表示 |
| 高頻度 CC による発行過多 | 誤操作・負荷 | デッドゾーン/レート制限/相対エンコーダ化で抑制 |
| アンチチート（ゲーム）による入力ブロック | 一部ゲーム不可 | スキャンコード送信・制約明記。サポート対象外と宣言 |
| キーロガー的挙動として AV 誤検知 | 配布時の警告 | 署名（コードサイニング）を将来導入、OSS 公開で透明性確保 |

## 8. 受け入れ基準（抜粋）

- [ ] MIDI デバイスを抜き差しすると、3 秒以内に UI のデバイス一覧へ反映される。
- [ ] ノブを回すと入力モニターに CC 値が流れ、対応インジケーターが動く。
- [ ] 「ラーン」でパッドを叩き Ctrl+Z を割り当てると、メモ帳上で Undo が発火する。
- [ ] Clip Studio Paint を前面化すると専用プロファイルに自動切替し、トレイ表示が変わる。
- [ ] 基本プロファイルに置いた Copy が全アプリで効き、個別プロファイルの上書きが優先される。
- [ ] 設定を JSON で保存・再読込しても割り当てが保持される。
